🖱️ Cursor-Based Function Cursors

cursor_passenger_booking_summary() opens an explicit PL/pgSQL CURSOR over the passengers table, fetches each row one at a time inside a LOOP, aggregates booking statistics per passenger, and returns a result set. This mirrors how application-level cursors work but entirely inside PostgreSQL.

OPEN passenger_cursor; LOOP FETCH passenger_cursor INTO rec; EXIT WHEN NOT FOUND; SELECT COUNT(*), SUM(fare_amount), MAX(booking_date) INTO v_bookings, v_spent, v_last FROM bookings WHERE passenger_id = rec.passenger_id; RETURN NEXT; END LOOP; CLOSE passenger_cursor;

🔗 Complex Joins Joins

This query joins 7 tables: bookings → passengers → schedules → trains → routes → stations (×2 for source/dest) → seats → payments. It demonstrates INNER JOINs for required relations and LEFT JOINs for optional ones (a booking may have no seat or no payment yet).

FROM bookings b JOIN passengers p ON b.passenger_id = p.passenger_id -- INNER: must have passenger JOIN schedules s ON b.schedule_id = s.schedule_id -- INNER: must have schedule JOIN trains t ON s.train_id = t.train_id -- INNER: must have train JOIN routes r ON s.route_id = r.route_id -- INNER: must have route JOIN stations src ON r.source_station_id = src.station_id JOIN stations dst ON r.destination_station_id = dst.station_id LEFT JOIN seats se ON b.seat_id = se.seat_id -- LEFT: seat may be null LEFT JOIN payments py ON b.booking_id = py.booking_id -- LEFT: payment may not exist

🔒 Transactions Transactions

Every booking creation, payment, and cancellation executes inside an explicit BEGIN / COMMIT transaction block. If any step fails, the entire operation is ROLLBACK-ed — no partial data is saved. The table below shows the outcome of past transactions.
